# AimHarder API Discovery

## Purpose and evidence boundary

This document evaluates whether the supplied **AimHarder Public API 1.0.0**
documentation contains the concepts required by the Business Intelligence
Copilot MVP.

Authoritative input: `experiment/input/AimHarder_API.pdf`, captured on
2026-07-18, 111 pages, SHA-256
`47f6d0ea156ca0f7186559818a0fc855a396c260043aa015aa1c01eab9fa457f`.

The PDF was inspected through text extraction and representative page renders.
No request was sent to AimHarder, no credentials were requested, and no example
payload was treated as observed runtime behaviour. Page references below refer
to the printed PDF page number.

## Authentication and transport

| Concern | Documented behaviour | Evidence | Confidence boundary |
|---|---|---|---|
| Base URL | `https://api.aimharder.com` | p. 1 | Some endpoint examples conflict; see contradictions |
| Protocol | HTTPS and mandatory HTTP/1.1; documentation warns HTTP/2 may produce 403 | pp. 1-2 | Documented only |
| Authentication | Bearer access token in `Authorization` | p. 2 | No token was obtained or tested |
| Token renewal | A refresh token renews an expired access token; expiry may surface as 410 | p. 2 | Lifetime is not specified and is not configurable |
| Credentials | Generated by an administrator under AimHarder configuration | pp. 1-2 | Explicitly excluded from this MVP |

The future adapter must be read-only, minimize requested fields, redact tokens
from logs, force HTTP/1.1 where the client permits it, and treat authentication
behaviour as unverified until a separately authorized integration exercise.

## Pagination and synchronization

- Paginated endpoints use an opaque `cursor` returned as
  `pagination.nextCursor`; iteration stops when `hasMore` is `false` (pp. 3-5).
- The documented page size is 100 items. The cursor must not be constructed by
  clients.
- The document says all paginated endpoints accept inclusive `id_from`; several
  endpoints also document `id_to` (pp. 5, 57).
- Cursor iteration is not a stable snapshot: new records inserted during a scan
  may or may not appear (p. 5).
- Booking history documents ID-range filtering, but no booking-date range
  filter. Calendar lookup is date-addressed (`/calendar/:date_str`).
- No deletion feed, updated-at cursor, snapshot token, or documented global
  booking-history endpoint was found.

A future ingestion process therefore needs deduplication by stable IDs, a
high-water-mark policy with overlap, reconciliation for changes/cancellations,
and evidence from real payloads before claiming complete incremental sync.

needs. A future adapter should collect only client ID, display-safe identifier,
active state/creation date, and the analytical booking fields.

## Capability matrix

| Business need | Required data | Documented support | Derivation | Classification |
|---|---|---|---|---|
| Average occupancy | Dated sessions, capacity, eligible bookings | Calendar supplies dated capacity; client histories supply bookings | Combine records using a verified session identity; compute booking count / capacity per session | **Derivable with integration gap** |
| Lowest occupancy classes | Class, time, session capacity and bookings | Same sources as occupancy | Group session occupancy by class and start time | **Derivable with business definition** |
| Cancellation rate by time | Session time, all bookings and cancellations | Client histories contain time and `cancellation_date` | Cancelled / all booking records grouped by time | **Derivable** |
| Active users without attendance for >21 days | Active clients and attended booking history | `/clients` documents active clients; history has `attended` | Find latest `attended=true`; `/clients/no-booking` alone is insufficient | **Derivable with business definition** |
| Occupancy period comparison | Two sets of session occupancy | Same sources as average occupancy | Apply the identical metric independently to both periods | **Derivable with integration gap** |

“Derivable” means the documented concepts are sufficient for the normalized
mock and plausible for a future adapter. It does **not** mean efficient or
verified against the real service.

## Material integration gap: complete booking enumeration

The supplied document exposes booking lookup by known ID and history per
client, but no center-wide booking list. A read-only adapter could provisionally
enumerate active clients and then paginate each client's history. That has four
material limitations:

1. It is an N+1 request pattern and may be expensive or rate-limited.
2. Histories for deactivated clients may be omitted if only `/clients` is used.
3. The history item does not document `schedule_id`; joining by class, day and
   time may be ambiguous. Fetching every booking by ID would add another N+1
   traversal.
4. Cursor iteration is not documented as snapshot-consistent.

The mock data source will therefore expose a stable normalized `sessionId`, but
that field is a **model requirement**, not a claim that AimHarder booking
history supplies it. A future adapter must prove a reliable join or request a
better upstream endpoint before production use.

## Documentation contradictions and defects

| Claim A | Conflicting evidence | Consequence | Runtime treatment |
|---|---|---|---|
| Base URL is HTTPS production | Examples use `http://api.aimharder.com` and `https://api.aimharderlocal.com` | Copying examples can select an insecure or nonexistent target | Use documented HTTPS base provisionally; verify in future integration |
| Every response uses `data`, `pagination`, `info` envelope (p. 6) | Several endpoint success examples show raw arrays, including calendar/classes/clients | One deserializer cannot be assumed | Accept neither shape as observed; future adapter validates both explicitly |
| Calendar `cancelled` is a `Number` | Example returns `false` | Type is inconsistent | Normalize defensively only after observed evidence |
| `waitlist_count` suggests a count | Description says waiting time, example is a string | Meaning is unclear | Do not use it in MVP metrics |
| Schedule schema names `assistant_limit` | Example uses `assistants_limit` | Capacity mapping is uncertain | Prefer dated calendar `limit`; retain mapping ambiguity |
| General token section lists common HTTP codes without 410 | Authentication section says expiry returns 410 | Error handling contract is incomplete | Treat 410 as documented special case, not verified behaviour |
| No-booking endpoint name implies an inactivity aid | Payload reports last reservation, not last attendance | Booking recency can misclassify no-shows | Use attended history for the MVP rule |

These contradictions are retained rather than silently reconciled.

## Approved business definitions for the mock

1. **Session occupancy:** confirmed, non-cancelled bookings divided by session
   capacity. Waiting-list and cancelled bookings are excluded. A period's
   average gives each valid session equal weight.
2. **Low occupancy:** average session occupancy grouped by class and start time,
   ordered ascending with deterministic tie-breakers.
3. **Cancellation rate:** cancelled booking records divided by all booking
   records in the same start-time group.
4. **Inactive user:** a currently active user whose latest attended session is
   more than the requested number of whole calendar days before the fixed
   analysis date. For never-attended users, creation date is an explicit proxy.
5. **Comparison:** calculate the same occupancy definition for both periods;
   report percentage-point and relative differences plus comparability flags.

These are project business rules, not undocumented claims about AimHarder.

## Exclusions and limitations

- Cancelled class sessions and sessions with missing/non-positive capacity are
  excluded from occupancy metrics and counted in the explanation.
- The documentation does not define whether `state=confirmed` remains on a
  cancelled booking; normalized mock records carry an explicit cancellation
  timestamp and the metric prioritizes that fact.
- Attendance and reservation occupancy are different concepts. Occupancy uses
  reservations; inactivity uses actual attendance.
- Time zone is the box time zone. The mock fixes it to `Europe/Madrid` and uses
  date-only arithmetic for day thresholds.
- Multi-box identity, cross-midnight sessions, wait-list promotion history,
  retroactive edits and deleted records are outside this MVP.
- Real API completeness, rate limits, permissions and payload shapes remain
  impossible to verify without separately authorized access.

## Discovery outcome

All five MVP questions can be answered deterministically from a simulated
normalized dataset compatible with documented AimHarder concepts. The future
real adapter remains feasible in principle, but cannot be promised without
resolving booking enumeration, session joins, payload contradictions and
incremental synchronization through official clarification or authorized
runtime evidence.

This result is sufficient to proceed with the mock architecture. It does not
authorize or imply real AimHarder integration.
